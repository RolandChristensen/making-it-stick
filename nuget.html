Notes On Nuget

NuGet: MS .NET Package Management

The way to share useful code when using MS .NET is through NuGet.
Search the NuGet.org web site for useful code to avoid re-inventing the wheel.
A NuGet package is a zip file with the extension *.nupkg containing all the files needed and a descriptive manifest.
A package can be shared publicly (nuget.org) or privately on a private server, private cloud, private shared filesystem or even locally. You can configure which hosts can be accessed by any given computer to ensure packages are retrieved from a specific source.
The creator of a package builds and packs the source into a package then publishes the package on the host for consumers to search for and install in their project.

Package Target Compatibility

Compatibility means that the package was built for at least one target .NET framework. The consumers target framework should match, or be within the range of, the package target. For maximum compatibility, target the .NET Standard, for all .NET and .NET Core projects to consume. Currently (2/20/2023), target netstandard2.0 for the broadest set of APIs.

If you need APIs outside of .NET Standard (multi-targeting), create separate assemblies for the different target frameworks supported and include them in the same package. NuGet is smart enough to extract only the assemblies needed for the target framework. This is harder to maintain for obvious reasons.

NuGet Tools

  • dotnet CLI: Provides some NuGet CLI capabilities in the .NET Core tool chain. Installed with any .NET Core related workflows. Works for any SDK-style project.
  • nuget.exe CLI: Provides all NuGet capabilities. For use in creation and consumption of packages. This is a platform agnostic tool that does not interact with Visual Studio. Works for any non-SDK-style project. Download from nuget.org and place it in the path for PowerShell (C:\Windows\system32)
  • Package Manager Console: Visual Studio on Windows PowerShell commands for installing and managing packages.
  • Package Manager UI: Visual Studio on Windows UI for installing and managing packages.
  • Manage NuGet UI: Visual Studio Mac UI for installing and managing packages.
  • MSBuild: provides the ability to create packages and restore packages used in a project directly through the MSBuild tool chain.

Dependency Resolution

NuGet handles dependencies automatically. A package can have a dependency on various other packages and multiple packages can have the same dependency. Only one version of each package can be used and NuGet resolves which version is used.
If you need a higher version of a package, but another package uses a lower version, you are stuck with the lower version.

Tracking References to Packages for Easy Restoration

NuGet tracks what packages are installed and we check that into source control rather than check in binaries which would increase the footprint on source servers and local environments. When built the packages are restored from the hosts referenced. Build servers perform a NuGet restore step. Locally, you can use nuget restore on the NuGet CLI, dotnet restore on the dotnet CLI, or Install-Package in the Package Manager Console. Visual Studio is smart enough to automatically restore packages when building a project (unless you were dumb enough to disable this in settings :).
References are maintained in one of two formats:
PackageReference: (NuGet 4.0+) Maintains a list of top-level dependencies directly in your project file, so no second file is needed. An associated file "obj/project.assets.json" is generated dynamically to manage the overall dependencies including downstream. .NET Core projects always use this method.
packages.config: (NuGet 1.0+) An XML file that maintains dependencies. All packages installed are stored in "packages" folder. This is kept for backwards compatibility only.
Note: package references are only added when installing packages from the Visual Studio Package Manager (UI or Console) or dotnet.exe CLI. Do not use the nuget.exe if you expect the package to be installed by others downloading your source.
Example project reference in *.csproj file: 

  <ItemGroup>
    <PackageReference Include="package name" Version="#.#.#.#" />
  ...

NuGet Maintains Specifications for Package

Localization and Debug Symbols and how they are referenced including version ranges and pre-release versions are all managed by NuGet. It also provides APIs to work with its services programmatically for Visual Studio extensions and project templates.

Usage

  1. Search for and find useful packages that enable you to spend time on the code you really care about.
  2. Use NuGet to install the package.
  3. Include a using {namespace} statement to include the package in a class. The namespace is usually the same as the package name, which is a good idea if you are creating packages.

Tools

Guidelines
For developing on Windows or Mac, use Visual Studio to search for, install, and update packages. It will automatically add packages to the project files for source control.
If you are developing on multiple platforms, learn to use the Nuget CLI. There are limitations running under Mono on Linux and Mac, but still boasts the most features.
NuGet CLI on Windows has the most features available to it, with the limitation that it does not add package info to the project files (Use Visual Studio to install and update).

  • dotnet.exe CLI: automatically installed with any .NET Core related workloads in Visual Studio 2017+ and with the .Net SDK. Works on Windows, Mac, and Linux (No need to install Mono). To install on build servers you will need to use a tool for Continuous Integration.
  • nuget.exe CLI (Windows): Provides all NuGet functionality on Windows and most functionality on Mac or Linux running under Mono. Microsoft has nuget.exe downloads available, just search for it. Nuget.exe on Windows has the most features available to it, more than any of the others in this list. Note: do not use it to install packages, however, because it does not add references in project files, so it is best to use dotnet.exe or Visual Studio.
  • nuget.exe CLI (Mono):
  • Visual Studio: most features are available in Package Manager Console and in Package Manager UI, with a pretty front-end. The commands available in Package Manager Console are only available via Visual Studio and not other PowerShell environments. Visual Studio (2017+) installer has an option to install NuGet.
  • Visual Studio for Mac: not all features are available. Use dotnet.exe or nuget.exe.
  • Visual Studio Code: NuGet functionality through extensions.
  • MSBuild CLI: for build servers you can restore and create packages, but is not really a general purpose tool for NuGet.

CLI tools: use dotnet.exe for .NET Core or .NET Standard. However, if you have a .NET Framework (non-SDK-style) project use nuget.exe unless the project has been migrated from packages.config to PackageReference. You can use dotnet.exe on Linux and Mac without installing Mono. Nuget.exe requires the installation of Mono on Linux and Mac.

Package references are included in the *.csproj file as shown below, with the caveats mentioned above when using NuGet CLI.

Create Packages

To create a package you will need to add metadata so the NuGet package manifest is created.

  1. Open the .csproj (or project file for f# or visual basic)
  2. Find the <PropertyGroup> tag and add the following XML
  3. <PackageId>unique-identifier.##.##.##.###.Test</PackageId> - This must be unique across nuget.org and other package sources.
  4. <Version>1.0.0</Version>
  5. <Authors>names</Authors>
  6. <Company>company</Company>

Add additional tags as needed. Search for MSBuild reference for .NET SDK projects to find additional information.

dotnet pack -h for help (--help is also valid)

dotnet pack - builds a .nupkg file from the project. The output of the command shows you where the package is located.

To automatically create the package each build, add <GeneratePackageOnBuild>true</GeneratePackageOnBuild> to the <PropertyGroup> of the project file.

dotnet pack --include-symbols adds debug symbols, so you can trace through the code.
dotnet pack --include-source adds a src folder to the package.

By default, dotnet pack, builds the project first, but may not be desireable in a CI build. Pass the --no-build option to avoid this.

UI Panel Integrated with Visual Studio

To open the NuGet Package Manager from Visual Studio.

  1. Using the menus: Project > Manage NuGet Packages...
  2. Using the right click menu: Right click the project in Solution Explorer then select Manage NuGet Packages...

Search Packages

From the NuGet Package Manager UI

  1. Select the "Browse" tab
  2. Select the package source from the dropdown (generally defaulted to nuget.org)
  3. Type the package to search for into the search box. (If you know what you are looking for this is easy, but it may be best to search using your browser before starting in the NuGet Package Manager.)

Install / Uninstall Packages

From the Package Manager UI:

  1. Search for and find the package to install
  2. Select the package to install in the search results
  3. Select the version (The latest is selected by default)
  4. Click the Install button
  5. A prompt may appear with additional dependencies you will need to install. Select OK, if you are happy with those.

From the NuGet Package Manager Console:

  1. Select the source from the dropdown.
  2. Select the project from the Default project dropdown.
  3. PM> Install-Package {package name}

Private Feeds

Use private feeds to keep your private code private and to restrict third-party libraries developers can use.

  • Local Feed: Packages are placed on a network share. Use nuget init and nuget add to create a hierarchical folder structure.
  • NuGet.Server: Local HTTP server
  • NuGet Gallery: Hosted on an Internet server using NuGet Gallery Project, which provides user management and features similar to nuget.org

There are a host of other NuGet hosting products out there. A simple search will uncover: Artifactory, Azure Artifacts, BaGet, Bytesafe, Cloudsmith, Gitea, GitHub package registry, GitLab Package Registry, JetBrains Space, LiGet, MyGet, Nexus Repository OSS, NuGet Server (Open source), ProGet, Sleet, and TeamCity.

Add the available sources to the NuGet.Config file using Visual Studio or from the nuget CLI.

Local Feeds

A hierarchical folder structure on the local network or local computer containing packages. These are then used as package sources in your favorite NuGet manager.
See instructions above for your favorite Package Management tool to add the source. In this case simply add the folder path.

Initializing and Maintaining Hierarchical Folders

Older versions of NuGet did not maintain hierarchical folders and instead put all packages in a single folder, decreasing performance.
The folder hierarchy follows this structure: 

\\serverName\packages
  packageID
    versionNumber
      packageID.version.nupkg
      file1
      file2
      ...
To add a *.nupkg file to a source use the nuget command
nuget add new-package.1.0.0.nupkg -source \\serverName\packages - This will set up the folder structure necessary on the server.
nuget init c:\packages \\serverName\packages - Sets up an entire folder of packages as if using the add command repeatedly.
